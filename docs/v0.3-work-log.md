## 阶段 2：DataContext 和隐私 — ✅ 完成（2026-07-16）

### 完成内容
- **`lib/semantic/detect-sensitive.ts`**：敏感字段检测（字段名关键词 + identifier 角色 + 值模式手机/邮箱/身份证/银行卡）+ 稳定掩码（保留长度、首尾可见、同值同掩码的一致映射，纯函数无 `Math.random`/`crypto`）。
- **`lib/semantic/build-data-context.ts`**：构建 `DataContext`（SPEC 9）：
  - 确定性哈希（djb2）+ PRNG（mulberry32），种子 = `hashStr(datasetId + rowsHash)`，保证同数据同采样。
  - 采样：头 8 / 中 6 / 尾 6 / 随机 12，按行 index 去重，上限 40；`boundaryRows`（数值 min/max/p25/p75）、`anomalyCandidateRows`（IQR，≤10）。
  - 列上下文：复用 `computeNumericStats`/`computeCategoryStats`，只含客观数据（`detectedType`/`heuristicHints`/`numericStats`/`topValues`），不写业务结论。
  - 脱敏：发给 LLM 的样本经 `maskRow`；数值统计用原值（不因掩码改变）。
  - `tokenBudget`（4 字符/token 估算）+ `omittedSections`（关闭行样本时记录）。
  - 提示注入防护：单元格值仅作数据保留，不改变结构（SPEC 9.5）。
- **测试**：`tests/sensitive-mask.test.ts` + `tests/data-context.test.ts`（覆盖规格 24.2 矩阵：小表全量、大表 ≤40、头中尾覆盖、稳定性、不同 hash、脱敏、高基数裁剪、token 截断、注入防护、不含业务结论）。

### 设计决策
- 不复制统计逻辑：直接复用 `lib/analysis/statistics`、`lib/analysis/profile`（SPEC 13.1「不得复制两套相同计算」）。
- `truncated` 语义 = 原始行数 > 40（数据多于能发送的样本量），清晰反映采样信息有损。
- `DataContext.quality` 在旧数据缺失时兜底构造；`generatedAt` 随构建时间（采样/统计仍稳定）。
- 修复潜在栈溢出：日期极值用循环求 min/max，不用 `Math.min(...bigArray)`。

| `tsc --noEmit` | ✅ EXIT 0 |
| `vitest run` | ✅ **375 用例 / 24 文件**（阶段 1 后 348 + 阶段 2 新增 27） |

## 阶段 3：LLM 数据理解 — ✅ 完成（2026-07-16）

### 完成内容
- **`lib/semantic/understanding-prompt.ts`**：`UNDERSTANDING_SYSTEM_PROMPT`（SPEC 10.4 的 14 条 + SPEC 9.5 提示注入防护文本）+ `buildUnderstandingInput`（从已脱敏 DataContext 序列化字段统计/取值/样本/质量）。
- **`lib/semantic/understand-dataset.ts`**：`understandDataset` 服务：
  - LLM 未启用 → `fallback`，不产生伪 understanding（SPEC 21.1）；
  - `chatJSON` + `validateLLMUnderstanding` + 最多 2 次 JSON 修复（SPEC 4.2，网络/超时错误不修复）；
  - blocking ambiguity → `needs_user_input`，否则 `ready_for_confirmation`；
  - 失败 → `failed` + `understanding=null`。
- **`lib/semantic/apply-understanding.ts`**：用户修正合并（`applyFieldUnderstandingChanges`/`resolveAmbiguity`/`confirmUnderstanding`/`hasUnresolvedBlocking`），只改业务语义不覆盖物理类型。
- **`lib/schemas/understanding.ts`**：新增 `LLMUnderstandingSchema`（omit 服务端字段 `id/datasetId/createdAt/confirmedAt`），区分 LLM 输出校验与持久化校验。
- **`lib/store.ts`**：新增 `saveUnderstanding`/`getUnderstanding`/`saveContext`/`getContext`（原子写入 `understanding.json`/`context.json`，SPEC 19.1）。
- **API**：`POST /api/datasets/[id]/understand`（SSE：stage→understanding|ambiguity→done|error，复用已有理解）+ `GET/PUT /api/datasets/[id]/understanding`（读取 / 合并用户修正 + 可选 confirm，blocking 未解除时 422）。
- **测试**：`tests/understanding-service.test.ts`（fallback/成功/blocking/修复成功/修复失败/超时/存量识别/actual-target/用户修正）+ `tests/prompt-injection-data.test.ts`（注入文本作数据、system prompt 固定安全、无泄露）。

### 设计决策
- LLM 输出与持久化分离：`LLMUnderstandingSchema` 校验 LLM 返回（无服务端字段），补全后成完整 `DatasetUnderstanding`；`PUT` 合并后再过完整 schema 校验才落盘。
- `hasBlockingAmbiguity` 参数放宽为 `Pick<DatasetUnderstanding,"ambiguities">`，兼容 LLM 解析中间态。
- understand route 非 force 时复用已存理解，避免重复 LLM 调用。

## 阶段 4：理解确认 UI — ✅ 完成（2026-07-16）

### 完成内容
- **`lib/api-client.ts`** 扩展：`getUnderstanding` / `updateUnderstanding` / `runUnderstand`（SSE：stage→understanding|ambiguity→done|error，done.status=fallback 引导本地模式）+ `FieldUnderstandingChange` 类型。
- **5 个新组件**（SPEC 20.1）：
  - `UnderstandingStatus`：理解流程 UI 阶段徽章（idle/loading/ready/needs_input/confirmed/failed/fallback）。
  - `UnderstandingOverview`：展示数据集类型/结构/行粒度/业务描述/行含义/字段关系/建议目标（只读）。
  - `FieldUnderstandingTable`：字段语义可编辑（role/measureBehavior/subRole/recommendedAggregation/businessMeaning），实时上报修改。
  - `AmbiguityPanel`：blocking ambiguity 高亮 + 提示必须处理。
  - `DerivedMetricSuggestions`：派生指标建议（需确认的标注）。
- **预检页 `/import/[draftId]` 改造**：在质量概览后插入「AI 数据理解」卡片——开始理解 / 显示结果 / 字段语义修正 / 保存 / 确认 / 重新理解；fallback 显示本地模式引导；blocking 未解除时禁用确认；加载时复用已存理解（SPEC 19.3）。

### 设计决策
- 理解区域为**可选增强**，保留现有字段物理配置表（FieldConfigTable）与 confirm 流程，零破坏。
- 用户修改乐观更新本地视图 + 累积 `pendingChanges`，保存/确认时 PUT 落盘（SPEC 10.5）。
- 「未确认理解阻断默认 LLM 分析」的强制在阶段 8 analyzer 门面（`requireConfirmedUnderstanding`）落实；阶段 4 仅实现 UI 与交互。
